Let's walk through this diagram. At the top, filesystem tools: read_file, write_file, ls, and so on, all flow down through the Backend. The backend is what you configure to control where files actually live and where commands actually run.

[Point to State]

The default is State. Files are kept in the agent's in-memory state. Nothing persists to disk, which is fine for simple use cases or demos.

[Point to Disk]

Filesystem backend persists files to local disk. Good for local development where you want files to actually exist on your machine.

[Point to Store]

Store connects to a remote storage backend, useful when your agent is deployed and needs persistent file access across runs.

[Point to Context Hub]

Context Hub is a managed backend that handles file storage in a hosted environment.

[Point to Sandbox and LocalShell, and the execute tool node]

We will cover Sandboxes in a later lesson.

[Point to Composite and the Router]

Composite is the most flexible option. It routes file operations to different backends depending on the path prefix, so you could have memory files going to one backend, working files to another, and so on. All within the same agent.

[Point to Custom]

Finally, Custom lets you implement your own backend if none of the built-ins fit your needs.

[Scroll to "The Interpreter"]

The interpreter is separate from all of this. Language models are unreliable at exact computation (such as calculations, data transformations and generating precise output) so rather than having the model try to reason through those, it writes code and the interpreter actually runs it, returning the real result. It runs as a tool in the agent's main loop, not through a backend. The last lesson of this module covers it in detail.

The key idea to take away from this lesson is that the backend is what makes the execution environment configurable. Swap it and you change where files live and where commands run, without changing anything else about the agent.

A Deep Agent has an **execution environment** that provides three things:

- a **filesystem** to read and write files
- a **shell** to run commands (the `execute` tool)
- an **interpreter** to run code

The first two run through a pluggable **backend** (below); the interpreter runs as a tool in the main loop.

Every filesystem and `execute` call passes through whichever backend you configure. The default keeps files in the agent's **state**; others persist them to local **disk** or a **store**, run them in an isolated **sandbox**, or send files to different backends by path prefix (a **composite**). Sandbox and local-shell backends also add the `execute` tool. The next two lessons cover each in detail.

## The interpreter

The interpreter lets the agent run code. Language models are unreliable at exact computation, so when a task needs a precise result - a calculation, a data transformation, a generated file - the agent writes code and the interpreter runs it, returning the actual output. It runs as a tool in the agent's main loop, alongside the agent's other tools. The final lesson of this module covers it in detail.

---

## Recap

- The execution environment gives the agent three capabilities: a **filesystem**, a **shell** (`execute`), and an **interpreter**
- The filesystem tools and `execute` all run through a swappable **backend**: swap it to change where files live and where commands run, without touching the agent
- The **interpreter** is the exception: it runs as a tool in the main loop, not through a backend
