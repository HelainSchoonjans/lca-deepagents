MCP: Connecting Agents to External Services

Model Context Protocol (MCP) is an open protocol that standardizes how applications provide tools and context to LLMs. Instead of each integration requiring custom code in your agent, MCP servers expose a consistent interface that any MCP-aware client can connect to.

LangChain agents connect to MCP servers using the langchain-mcp-adapters library. The adapters wrap each MCP tool as a standard LangChain BaseTool, so the agent sees it identically to any locally-defined tool. The tool result is added to the message history and the model decides what to do next, exactly as it would with a regular tool call.

Configuration

deepagents discovers MCP servers from .mcp.json in your project root (or ~/.deepagents/.mcp.json for user-level config). The project repo already includes a config pointing at the LangChain documentation server:

Load those tools in Python with get_mcp_tools:

# python/m1/m1.6_scratch_agent_mcp.py
from deepagents_code.mcp_tools import get_mcp_tools

tools, session_manager, server_infos = await get_mcp_tools(".mcp.json")

Pass tools to create_deep_agent as you would any other tool list. When you are done, call await session_manager.cleanup() to close the connections.

Transport details

The transport is the underlying mechanism used to communicate with an MCP server. It is abstracted away from your agent code; the agent never sees or interacts with it directly.

There are three transport types:

  • stdio runs the server as a local subprocess on your machine. Messages pass as JSON through the subprocess's standard input and output streams. No networking required.
  • HTTP sends a standard HTTPS request to a remote server and waits for the response. A new connection opens for each tool call and closes when the response arrives.
  • SSE (Server-Sent Events) opens a persistent HTTP connection. The server can push events back over that connection in real time, which is useful for long-running tools or streaming results. SSE is the newest transport type in the MCP spec.

In .mcp.json, set "type" to "http" or "sse" for remote servers, or use "command" and "args" for stdio servers. deepagents normalizes the transport at load time; your Python code does not change.

Authorization

Some MCP servers require the user to authenticate before the agent can call their tools.

Add "auth": "oauth" to the server entry in .mcp.json:

// .mcp.json
{
  "mcpServers": {
    "gmail": {
      "type": "http",
      "url": "https://gmailmcp.googleapis.com/mcp/v1",
      "auth": "oauth",
      "allowedTools": ["search_threads", "get_thread"]
    }
  }
}

When deepagents detects a server that requires OAuth, it pauses and opens a browser window for the user to log in. Tokens are stored in ~/.deepagents/.state/mcp-tokens/ and reused on subsequent runs.

allowedTools lets you restrict which tools from a server the agent can see. This is useful when a server exposes many tools but you only need a subset; for example, giving an agent read-only access to Gmail without exposing tools that can send or delete.

Recap

  • MCP is an open protocol that standardizes how applications expose tools and context to LLMs; any MCP-aware client connects to any MCP server without custom integration code
  • langchain-mcp-adapters wraps each server tool as a standard BaseTool; the agent cannot tell MCP tools apart from locally-defined ones
  • Servers are configured in .mcp.json; the type field sets the transport (stdio for local subprocesses, http for stateless remote calls, sse for persistent streaming connections)
  • allowedTools restricts which tools from a server the agent can see; useful when a server exposes many tools but you only need a subset, or want to limit access to read-only operations
  • auth: "oauth" triggers a one-time browser authorization on first run; tokens are cached in ~/.deepagents/.state/mcp-tokens/ and reused automatically

    "mcpServers": {
        "docs-langchain": {

allowedTools vs. disabledTools

allowedTools is an allowlist; only the named tools are kept; anything new the server adds in the future stays invisible. disabledTools is a denylist; only the named tools are removed; anything new the server adds is visible by default. Use allowedTools when you want a strict, stable subset; use disabledTools when you want most of the server's tools and just need to block a few.

MCP with OAuth

🎥 Video Walk-through [click to expand]


Many of the most useful tools and services require a user to be signed in to access data. Increasingly, application vendors offer MCP interfaces that use the OAuth standard to allow an agent to gain access.

We will use the GitHub MCP server as our example, as most students will have a GitHub account. It requires an OAuth access token before it will respond to any request. This lesson covers two things: how to set up your agent to obtain and use that token, and what actually happens under the hood when it does. It will demonstrate static client registration, where you manually register the client with the provider once before running any code. This is very common at this point in time.

Set up your agent with OAuth

Two steps are required:

  1. Register your OAuth application — a one-time setup.
    - This identifies your agent to the application. It can also be a time
    to establish constraints on access in the application.
  2. Configure your agent with the credentials and run it.

Register your OAuth client

Before running any of the programs below, you need to register an OAuth App
in GitHub. This is a one-time manual step — it is what makes GitHub willing to
issue tokens to your application at all.

Follow GitHub's guide: Creating an OAuth App.

When filling in the registration form at
github.com/settings/developers → New OAuth App, use these values:

Field Value
Application name LCA-DA-Course-MCP
Homepage URL http://localhost
Authorization callback URL http://127.0.0.1

The callback URL http://127.0.0.1 (an IP literal, not a hostname) is the
loopback address the script's listener binds to. GitHub validates the host but
not the port for loopback addresses, so the script can pick an available
ephemeral port at runtime without pre-registering a specific one.

After clicking Register application, immediately click Generate a new
client secret — GitHub shows it only once. Add both values to .env:

# .env
GITHUB_CLIENT_ID=your_client_id_here
GITHUB_CLIENT_SECRET=your_client_secret_here

GITHUB_CLIENT_ID is used to build the authorization URL during the OAuth
consent flow. GITHUB_CLIENT_SECRET is used to exchange the authorization
code for an access token.

Putting it together (github_agent.py)

This example follows the template shown here.

# python/m1/gh_auth/github_agent.py
"""GitHub MCP agent — OAuth via the MCP SDK's built-in flow.

The MCP SDK's OAuthClientProvider handles PKCE, discovery, and token exchange.
GitHubTokenStorage pre-seeds the registered client credentials so the SDK skips
dynamic client registration (which GitHub does not support).

Run:  uv run python m1/gh_auth/github_agent.py
"""

from __future__ import annotations

import asyncio
import os
import threading
import webbrowser
from http.server import BaseHTTPRequestHandler, HTTPServer
from pathlib import Path
from urllib.parse import parse_qs, urlparse

from deepagents import create_deep_agent
from dotenv import load_dotenv
from langchain_mcp_adapters.client import MultiServerMCPClient
from langchain_mcp_adapters.tools import load_mcp_tools
import httpx
from mcp.client.auth import OAuthClientProvider
from mcp.shared.auth import OAuthClientInformationFull, OAuthClientMetadata, OAuthToken

from models import model

ENV_PATH = Path(__file__).resolve().parent.parent.parent / ".env"
load_dotenv(dotenv_path=ENV_PATH, override=True)

TOKEN_FILE = Path(__file__).resolve().parent.parent.parent / ".m1_github_token"

MCP_URL = "https://api.githubcopilot.com/mcp/"
REDIRECT_URI = "http://127.0.0.1:8765/"
SCOPE = "repo read:user"


class GitHubOAuthClientProvider(OAuthClientProvider):
    def __init__(self, *args, requested_scope: str, **kwargs) -> None:
        super().__init__(*args, **kwargs)
        self._requested_scope = requested_scope

    async def _perform_authorization_code_grant(self) -> tuple[str, str]:
        self.context.client_metadata.scope = self._requested_scope
        return await super()._perform_authorization_code_grant()

    async def _exchange_token_authorization_code(
        self, auth_code: str, code_verifier: str, *, token_data: dict | None = {}
    ) -> httpx.Request:
        req = await super()._exchange_token_authorization_code(
            auth_code, code_verifier, token_data=token_data
        )
        req.headers["Accept"] = "application/json"
        return req


class GitHubTokenStorage:
    def __init__(self, client_id: str, client_secret: str) -> None:
        self._client_id = client_id
        self._client_secret = client_secret

    async def get_tokens(self) -> OAuthToken | None:
        if TOKEN_FILE.exists():
            raw = TOKEN_FILE.read_text().strip()
            if raw:
                try:
                    return OAuthToken.model_validate_json(raw)
                except Exception:
                    return OAuthToken(access_token=raw, token_type="Bearer")
        return None

    async def set_tokens(self, tokens: OAuthToken) -> None:
        TOKEN_FILE.write_text(tokens.model_dump_json())

    async def get_client_info(self) -> OAuthClientInformationFull:
        return OAuthClientInformationFull(
            client_id=self._client_id,
            client_secret=self._client_secret,
            redirect_uris=[REDIRECT_URI],
            token_endpoint_auth_method="client_secret_post",
        )

    async def set_client_info(self, info: OAuthClientInformationFull) -> None:
        pass


async def open_browser(url: str) -> None:
    print(f"Opening browser for GitHub login ({SCOPE})...")
    print(f"If it doesn't open, paste this URL:\n{url}\n")
    webbrowser.open(url)


async def wait_for_callback() -> tuple[str, str | None]:
    result: dict[str, str] = {}
    done = threading.Event()

    class Handler(BaseHTTPRequestHandler):
        def do_GET(self):  # noqa: N802
            params = parse_qs(urlparse(self.path).query)
            result["code"] = params.get("code", [""])[0]
            result["state"] = params.get("state", [""])[0]
            self.send_response(200)
            self.send_header("Content-Type", "text/html")
            self.end_headers()
            self.wfile.write(b"<h3>Authorized. You can close this tab.</h3>")
            done.set()

        def log_message(self, *a): pass

    port = int(urlparse(REDIRECT_URI).port or 80)
    server = HTTPServer(("127.0.0.1", port), Handler)
    threading.Thread(target=server.serve_forever, daemon=True).start()

    if not done.wait(timeout=300):
        server.shutdown()
        raise SystemExit("Timed out waiting for OAuth callback.")
    server.shutdown()
    return result["code"], result.get("state")


async def main() -> None:
    client_id = os.environ.get("GITHUB_CLIENT_ID")
    client_secret = os.environ.get("GITHUB_CLIENT_SECRET")
    if not client_id or not client_secret:
        raise SystemExit("Set GITHUB_CLIENT_ID and GITHUB_CLIENT_SECRET in .env.")

    auth = GitHubOAuthClientProvider(
        server_url=MCP_URL,
        client_metadata=OAuthClientMetadata(
            redirect_uris=[REDIRECT_URI],
            scope=SCOPE,
            token_endpoint_auth_method="client_secret_post",
        ),
        storage=GitHubTokenStorage(client_id, client_secret),
        redirect_handler=open_browser,
        callback_handler=wait_for_callback,
        requested_scope=SCOPE,
    )

    client = MultiServerMCPClient({
        "github": {
            "transport": "streamable_http",
            "url": MCP_URL,
            "auth": auth,
        }
    })

    async with client.session("github") as session:
        tools = await load_mcp_tools(session)
        print(f"github: {len(tools)} tool(s) available")

        agent = create_deep_agent(model=model, tools=tools)
        print("Running agent...\n")
        result = await agent.ainvoke({
            "messages": [{"role": "user", "content": (
                "Use the get_me tool to find my GitHub username, "
                "then use get_file_contents to list the files in the root of my repository."
            )}]
        })
        print("\n" + result["messages"][-1].content)


if __name__ == "__main__":
    asyncio.run(main())

OAuthClientProvider is from the MCP Python SDK and implements httpx.Auth — the interface httpx uses to inject authentication into every request automatically. It handles the full OAuth flow: it follows the WWW-Authenticate header to discover the authorization server, generates PKCE parameters, calls redirect_handler to open the browser, awaits callback_handler to receive the authorization code, exchanges the code for a token, and caches it via storage.

GitHubTokenStorage pre-seeds the registered client_id and client_secret so the SDK skips dynamic client registration, which GitHub does not support. GitHubOAuthClientProvider is a small subclass with two GitHub-specific fixes: it restores the requested scope (the SDK otherwise uses GitHub's full advertised scope list), and adds Accept: application/json to the token exchange request (GitHub returns URL-encoded form data by default).

On first run (no cached token) the program opens the browser for consent; on subsequent runs it reads the cached token and skips straight to the agent:

Under the hood

🎥 Video Walk-through [click to expand]


Strictly speaking, OAuth is an authorization framework — but the consent
flow bundles authentication in as well:

  • Authentication — the user proves their identity by logging in to the
    service. The agent cannot skip this; a human must be present to approve.
  • Authorization — the user grants the agent permission to act on specific
    scopes (for example, repo to read repositories). The token the agent
    receives is bounded by exactly those scopes and nothing more.

The OAuth flow involves five actors:

  • you / the user — who approves access in the browser;
  • the client (myagent, your script) — the thing that wants to call tools;
  • the browser — which carries you to GitHub to log in and returns with the
    authorization code;
  • the authorization server — GitHub, which logs you in and issues tokens;
  • the resource server — the GitHub MCP server, which holds the protected
    tools.
sequenceDiagram participant U as User participant A as myagent participant B as Browser participant O as GitHub OAuth
Server participant M as GitHub MCP
Server A->>M: Step 1 — Probe (no credentials) M-->>A: 401 + resource metadata → endpoints A->>B: Step 2 — Open browser (PKCE) B->>U: Consent screen U->>B: Approve B-->>A: authorization code A->>O: Step 3 — Exchange code for token O-->>A: access_token A->>M: Retry with Bearer token M-->>A: Tool result

Each step below is a standalone program that shows one leg of the sequence.

The WWW-Authenticate header in the 401 points to the protected-resource
metadata URL. Fetching it returns the authorization_servers and
scopes_supported fields shown in the output. GitHub publishes no RFC 8414
discovery document at that server URL, so the final two lines — the
authorization_endpoint and token_endpoint — are derived from the issuer
rather than looked up. Those two endpoints and the scopes are everything
Step 2 needs.

Two things happened, and each is worth separating.

PKCE protected the code in transit. Before opening the browser, the script
generated two values:

  • a code_verifier — a high-entropy secret it keeps in memory and never sends
    through the browser;
  • a code_challenge — the SHA-256 hash of the verifier, which it does send
    in the authorization URL (code_challenge_method=S256).

GitHub staples the challenge to the code it issues. In Step 3, redeeming the
code requires presenting the original verifier; GitHub re-hashes it and checks
it matches. The payoff: the authorization code travels through the browser and
a redirect URL — places it can leak — but a stolen code is useless without the
verifier, which never left the script.

The browser round-trip produced an authorization code. After you consented,
GitHub redirected back to a one-shot listener on 127.0.0.1 carrying the code.
The script also sent a random state value and checks that the redirect
returns the same one, which prevents a forged callback from injecting a code.

Note on the browser. The script calls webbrowser.open(), which uses
your operating system's default browser. If that opens the wrong account,
copy the URL printed after "If it doesn't open" and paste it into a browser
window where you are signed into the correct GitHub account.

The captured string — df2407422664... — is an opaque, short-lived,
single-use authorization code. It has no decodable structure; it is a handle
that only GitHub's token endpoint can redeem, and only when presented alongside
the code_verifier from PKCE. Step 3 performs that exchange.

The authorization code signifies that a real user approved the request. The
token itself is never sent through the browser — redirect URLs end up in
browser history and server logs, so the design keeps the token out of that
path entirely. The code crosses the browser boundary safely because it is
worthless without the verifier and client secret, which never left the script.

Step 3 — Exchange the code and retry (step3.py)

This step POSTs the authorization code to GitHub's token endpoint, receives an
access token, and retries tools/call with Authorization: Bearer.

sequenceDiagram participant A as myagent participant O as GitHub OAuth

The token exchange is a direct server-to-server POST — no browser involved.
The script sends the authorization code, the code_verifier from PKCE, the
redirect_uri (must match what Step 2 used), and the client_secret. GitHub
re-hashes the code_verifier against the code_challenge it stored when it
issued the code, and if they match, it returns an access token.

The access token — gho_TGboFbwQ... — is the credential that authorizes the
agent to act. The gho_ prefix identifies it as a GitHub OAuth token; it is
40 characters of opaque data, not a JWT. With it, the same requests that
returned 401 in Step 1 now succeed.

Step 2 saved the authorization code and code_verifier to a temp file
(.m1_github_state.json) so Step 3 could read them as a separate program.
Step 3 deletes that file immediately after the exchange — the code is
single-use and is now spent.

Recap

  • A protected MCP server gates every request behind a 401. That 401 is
    self-describing: its WWW-Authenticate header points to resource metadata
    naming the authorization server and scopes (RFC 9728).
  • Authorization servers vary in what they publish. GitHub does not publish
    an RFC 8414 discovery document; when discovery is absent, endpoints are
    derived from the issuer URL.
  • OAuth proceeds in two halves: get the user's authorization code (Step 2),
    then exchange it for an access token (Step 3).
  • PKCE binds the code to a secret the client never transmits, so a leaked
    code is worthless on its own.
  • Authorization codes and GitHub access tokens are opaque bearer handles;
    the server validates them by lookup, not by reading their contents.
  • An agent can manage its own OAuth end-to-end — probe, consent, cache, connect
    — without any manual steps between the phases.

Question: What does the RFC 9728 protected-resource metadata document provide?
A) The authorization server name and the scopes the resource accepts
B) The list of MCP tools available at the resource
C) A signed JWT the client must include in the next request
D) The user identity and session token
Answer: A ✅
The protected-resource metadata (fetched from the URL in WWW-Authenticate) names the authorization server and the scopes it accepts — the two values needed to start an OAuth flow.
Question: What attack does PKCE protect against?
A) Clock skew between the client and the authorization server
B) An expired client secret being replayed by an attacker
C) An attacker intercepting the authorization code and redeeming it before the client can
D) A forged access token submitted directly to the resource server
Answer: C ✅
PKCE binds the authorization code to a code_verifier that never leaves the client. An intercepted code is useless without the verifier — GitHub re-hashes it and rejects a mismatch at the token endpoint.
Question: Why does github_agent.py load MCP tools inside a client.session() context manager?
A) GitHub enforces a one-tool-per-connection limit
B) The context manager rotates the access token on every tool call
C) The context manager signs each request with the OAuth token
D) Tool calls require an active transport connection, which the context manager keeps open for the agent run
Answer: D ✅
MultiServerMCPClient.session() opens and holds a transport connection. Loading tools and calling them both require that connection; closing it between the two silently breaks tool calls.
Question: GitHub does not publish an RFC 8414 authorization-server discovery document. How does step1.py resolve the OAuth endpoints?
A) It hardcodes the endpoints as constants in the source file
B) It derives them from the issuer URL returned in the resource metadata
C) It asks the user to paste the endpoints from the GitHub settings page
D) It reads the endpoints from a GitHub OAuth App entry in the .env file
Answer: B ✅
The resource metadata names the issuer (https://github.com/login/oauth). Since GitHub publishes no discovery document at that base URL, step1.py appends /authorize and /access_token to construct the endpoints.
